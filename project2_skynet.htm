1. TEST SCOPE

SkyNet allows users to remotely monitor telescope systems across networks of observatories. The system makes use of interfaces for managing user permissions, and controlling scheduling, storage, maintenance and troubleshooting. It also allows users to view and export data recorded by the telescopes. The system relays task and data storage through a backend controller.

The SkyNet architecture is divided into four functional modules: the client, primary server, storage, and controller server tiers. The client tier provides the primary interface for users. SkyNet's controller server tier is directly responsible for the commanding the physical components of attached telescopes. Media is stored and accessed by the storage tier. These modules are all connected by the primary server tier, which coordinates the operations between them. 

Our test plan will examine SkyNet's four architectural modules introduced above. We will explain our approach for scrutinizing its quality through comprehensive testing in accordance with the developers' design outline, and discuss the relevant overhead infrastructure necessary for each task. Lastly, we will conclude with a validation testing framework and corresponding traceability matrix. 

2. TEST PLAN OVERVIEW

The test plan for SkyNet will follow its various tiers as they are being developed.  There will be four phases with each phase corresponding to each tier of the SkyNet system.  SkyNet, as the design document states, will have well defined interfaces in each tier and this will help tremendously to write integration tests against them.  In some cases, the integration tests written can target one or two tiers in isolation, but other tests may involve testing all tiers together.  It is important to design the integration test plan of the SkyNet system starting from the bottom tier up.  This means starting integration testing against the controller tier and working up to the client tier.  The client tier integration tests may test across all tiers and this is desired as all the other tiers should be in place by the time the integration testing on the client tier starts.  This system will require integration tests covering both software and hardware, with the hardware mostly being the components of the observatory.  There will be some third-party software to perform integration tests against such as the APIs used for component communication in the observatory and the APIs used for database and file system access.

2.1 Controller Tier Testing Phase

The first phase or integration build to test against will be the observatory controller tier build.  Once a build is obtained tests can be created to test the observatory controller tier application against all the mount, weather, and imaging related hardware the observatory controller tier application will be working with at time of release.  Not all the hardware will be available so there may need to be mocks setup against the APIs used to communicate with a given hardware device.  This will allow for some form of testing with those devices in the larger context of the operations the observatory controller tier implements.  There will need to be tests against the parts of the software controlling the mount and performing image acquisition as well.  This will involve testing the automation side of mount and image acquisition like ensuring the mount points the telescope at the proper target given in the two-line element (TLE) data.  Some may think this is simple but not all mounts will get a target centered in an imaging chip every time, actually very rarely really.

Some other examples of the integration tests needed will be to ensure that the observatory controller tier application performs a plate solve properly.  This involves using the imaging camera and other third-party software to calculate how much to tell the mount to move in right ascension or declination to center the target.  This operation must be bulletproof as a target that is not centered in the imaging chip is no good to the astronomer desiring the image data of it.  Tests for critical automation the observatory controller needs to provide like ensuring the observatory is closed and the telescope is parked if clouds or moisture is detected by the weather station hardware will need to be written as well.  Finally, there are the tests needed for the handling of schedules for maintenance and image acquisition.  These tests need to ensure schedules do not overlap and that the objects desired for image acquisition are indeed in the slew and tracking range of the mount during the time desired.

2.2 Primary Tier Testing Phase

The second phase of integration testing will be against the primary server tier application build.  In this build tests will need to target the primary server tier application API and its responses.  Mocks against storage tier stubs will be needed since that will layer will not be ready yet.  Mocking of operations against the storage tier stubs will allow testing to simulate different scenarios the storage tier may provide, both good and bad.  There will also need to be mocking against stubs for the e-mail server interaction as well since it may not be configured in time either.

Some of the API operations we will test will be creating managing storage, managing users, adding new components, reviewing logs, scheduling observation and maintenance tasks, and troubleshooting components.  There will need to be tests ensuring that e-mail notifications are sent out when needed.  Since the controller tier build should be available at the time the primary server tier application tests are started there should be no need for stubs to its API interface.  Testing can be done against an instance of it directly for any primary server tier operations that interact with it.

2.3 Storage Tier Testing Phase

The third phase of integration testing will be the storage tier.  There will need to be tests from the storage tier repository library against the file system and database management system SkyNet will utilize.  It would be best if the file system and database were ready when integration testing starts but if they are not the interfaces to the file system and database libraries used can be mocked and their operations can be simulated if needed.  Ideally if the need for mocking were present there would be mocks for both good and bad conditions.  It does not make sense to mock everything going right all the time, there will need to be mocks for exceptions that can occur in the database and file system libraries for example.

Integration tests for the file system will include creating\deleting volumes and archiving of imagery data.  Tests for the database will include CRUD operations for users and components.  There will need to be integration stress testing against the database as well since there is a requirement to support 2000 users.  This can be accomplished with multi-threading while accessing the database and file system concurrently.  Once the database and file system is in place we will need to remove our stubs and mocks to the database and file system libraries and run the tests against the actual implementations of each.  It should be clearly stated that neither the database or file system implementations need to be in place to start integration testing of the storage tier, they could come at a later time if needed.      

2.4 Client Tier Testing Phase

This final phase of integration testing will be against the client tier application.  Most of this tier is UI components but it is important to note that this where all the tier applications of the SkyNet system can be integration tested together.  From the client tier it interacts with the primary server tier and the primary server tier interacts with the storage tier and observatory controller tier applications.  All of the other tier builds of the SkyNet system should be ready by the time the integration testing begins on the client tier build so very little stubbing or mocking will be needed.

The initial focus of integration testing will be the UI components involved in managing users, managing components, scheduling, and managing storage volumes and image files.  After this the integration testing will focus on ensuring as much of the other tiers are being tested through the UI components.  This will be tough but it is possible.  For example, to test the observatory controller tier application from the client tier application one would create a test where a scheduled imaging run is created for a certain target with certain components and a certain time range.  The test application would wait for the scheduled imaging test to execute and ensure that it was successful.  Later the test application would check the storage volume via the UI of client tier to ensure the images were successfully pushed to the storage volume as they should be.  This is only one example of how it is possible to integration test from the top most layer, the client tier, to the bottom most layers, the controller and storage tiers.

Guiding Control

Ensure that the guiding control portion of the observatory controller properly handles temporary loss of star mass on its guide star and that it is not bother by satellites, planes, or other temporary obstructions to the star being guided on.

Imaging Control

Ensure a sub can be taken for the given exposure time specified.

Imaging Control

Ensure a sub can be taken for the given quality specified.

Imaging Control

Ensure a sub can be taken for the given level of binning specified.

Imaging Control

Ensure the filter wheel will change to any filter when commanded to.

Imaging control

Imaging control

Ensure that before every sequence of subs is started that a focusing routine is started by the observatory controller and the resulting focus of the frame is acceptable.

Imaging control

Ensure that after every sub exposure is taken that if necessary the observatory controller starts a focusing routine to refocus the frame.

Automation

Ensure scheduled imaging runs execute as they should for different equipment configurations and targets.

Automation

Ensure scope\mount is in parked position and shutdown for any maintenance scheduled tasks.

Automation

Ensure if communication is lost between storage tier and/or primary server tier that scheduled tasks continue to execute.

Automation

Ensure if communication is lost between storage tier that images are copied correctly to the storage tier after communication comes back up.  No images can be lost.

Notifications

Ensure notifications are sent out via e-mail when a scheduled task begins and finishes.

Notifications

Ensure notifications are sent during imaging when elected to in the imaging scheduled task.  This encompasses getting notified when events happen during an imaging sequence like image capture started\completed, focusing, changing filters, or waiting for guiding to settle.
