2. HIGH-LEVEL DESIGN

The iPal’s primary starting point will be its Speech Processing Module (Section 2.1). Upon input, it will parse the command structure. It then sends the command to the Security Module (Section 2.5) to verify the user has the appropriate permissions. Upon success, it then identifies the necessary control flow, selecting one of two possible paths:

  1. If keywords pertaining to “Amazon” or “Google Home” are detected, the iPal will activate its Machine-Learning (Section 2.3) and Interoperability (Section 2.4) modules.
  2. If not, the command will be handled directly by the Speech Processing Module’s Siri Web Services Interface (Section 2.2).

The Machine-Learning Module makes use of a database to store inquiries and answers. It will also employ three machine-learning algorithms to enable fast, high-precision answer selection from Amazon or Google Home.

The Interoperability Module is the component directly responsible for querying and receiving answers from Amazon and Google Home services.

2.1 Speech Processing Module

The structure of our Speech Processing Module is displayed in Figure 3:


Figure 3: an overview of the iPal’s Speech Processing Module

SpeechProcessor primarily acts as the module’s command-and-control, coordinating the exchange of data between classes through the sending and receiving of event triggers. It will additionally communicate the resulting answers to the user.

WordListener employs Apple’s Siri API to identify keywords from a given input command.

InquiryBuilder prepares given keyword data into an abstract data type ready for processing, which we simply call an inquiry.

InquiryProcessor first identifies the appropriate data flow path given an inquiry; for inquiries containing keywords related to Amazon or Google Home, the module invokes the Machine-Learning and Interoperability modules; for all other cases, it passes the data to the Siri Web Services Interface. When completed, it will invoke AnswerBuilder.

AnswerBuilder will build an answer (our second abstract data type) and return it to SpeechProcessor.

Each class sends events signaling task completion to SpeechProcessor.

2.2 Siri Web Services Interface

We briefly introduce and discuss several of the key functionality features available in Siri Web Services below. These will form the basis for our Siri Web Services Interface to facilitate interaction with Siri-enabled applications.

Domains represent command categories that Siri Web Services can make sense of: audio, messaging, and payments. Intents map vocabulary to specific application actions. For example, if a user issues the command “Hey Siri, message Jill. I’m running late”, Siri would launch iMessage (the domain), and craft the message for Jill (the intent) after finding her in the user’s contact list.

Example Domain 1: VoIP Calling – The user will be able to initiate VoIP phone conversations using Siri-enabled applications. For example, Skype offers Siri integration, so placing a phone call with a person listed in the Skype’s contact list would be a supported operation.

Example Domain 2: Messaging – The iPal will also handle both incoming and outgoing messaging with Siri-supported applications. Users will be able to send texts, iMessages, and instant messages through the iPal.

Example Domain 3: Payments – Users will be able to send payments to other contacts through supported payment apps. If a user would like to send money to a friend or business, they could make use of the following command: “Hey Siri- Paypal John twenty dollars.” The iPal will surpass competing payment-supporting platforms making because of its robust security measures employing two-factor authentication and precise access control.

2.3 Machine-Learning Module

The iPal’s Machine-Learning module, shown in Figure 4 below, fulfills requirement R-7 from our requirements summary:


Figure 4: view of the iPal’s Machine-Learning Module

MachineLearning acts as the module’s coordinator. Additionally, it provides the interface enabling data flow to and from the other modules.

The module makes use of three machine-learning algorithms: DecisionTree, NaiveBayes, and NearestNeighbor.

When a new inquiry is sent from the Speech Processing Module, the MachineLearning class’s method predictAnswer()is invoked, which then calls each algorithm’s train() and predict() methods. 

The machine-learning algorithms compete against each other to find the highest accuracy.  If an algorithm makes a prediction with accuracy greater than 0.95, it is considered for the generation of the strings used during the creation of an answer object (of MachineLearningAnswer type). The probability score is returned by the algorithm's getProbabilityScore()method and is accessed after every call to an algorithm's predict() method. If no algorithm predicts with a probability score above 95%, the Machine-Learning Module will wait for an answer object to be provided by the Interoperability Module, as well as the keywords it was derived from.  It will then store this object for subsequent training runs.

2.4 Interoperability Module

The Interoperability Module’s diagram is displayed in Figure 5, along with the requirements it satisfies:


Figure 5: diagram of the iPal’s Interoperability Module

ServicesManager determines the necessary format for the inquiry, and passes the information along to WebService.

WebService - pushes the inquiry to the appropriate web service, and receives the resulting answer.

GoogleHomeInterface – interface for Google Home web services queries.

AmazonInterface – interface for Amazon web services queries.

AmazonAnswer – The answer object built specifically from any Amazon web service response.

GoogleHomeAnswer – The answer object built specifically from any Google Home web service response.

The Interoperability Module houses the code responsible for interacting with any service outside of the Siri Web Services framework. When iPal gets a request for Amazon or Google Home this module is responsible for communication with Amazon or Google Home to fulfill the request. The Speech Module will call the sendRequest() method on the ServicesManager class which will then determine which service interface, AmazonInterface or GoogleHomeInterface to send the request to. Both the AmazonInterface and GoogleHomeInterface classes implement the interface WebService, which contains the necessary method definitions used for interaction with each. The getKeywordIdentifiers() method of the WebService interface returns the list of keyword strings to look for in a user request to iPal. This allows the ServicesManager class to know which interface class to choose before a call to the sendRequest() method is made on either class. Finally the sendRequest() method returns an answer object of either type AmazonAnswer when making a sendRequest() method call against the AmazonInterface class or of type GoogleHomeAnswer when using the GoogleHomeInterface class. This answer object contains the response words to speak back to the user and the exact keywords that were used to generate the response words. This answer object is also sent to the Machine-Learning Module for storage for later training runs it makes.

2.5 Security Module

Shown in Figure 6 below, our security module satisfies requirements R-2 through R-5:


Figure 6: the iPal’s Security Module

user is our defined abstract data type denoting user identities and permissions. We allow administrators discretion on which Siri-based applications users have access to. For simplicity, user Amazon or Google Home permissions are either all or nothing.

serviceCredentials  contains username and password information for each user, as well as a Boolean denoting their ability to operate money-sensitive operations.

siriApp denotes a specific Siri-accessible application. This is the data type used in our permission lists. 

UserManager controls the entire module, coordinating authentication operations, and giving administrators the ability to add, remove, and modify users and their respective permissions.

AuthenticationService acts as a middleman for the authentication process, being invoked by UserManager.
