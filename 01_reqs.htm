1. REQUIREMENTS SUMMARY

In this section, we introduce and discuss our project's requirements, and conclude with a traceability matrix. Our intended customers are existing handsfree device users, or those wishing to expand their use of the Internet of Things. Our intended user audience includes the iPal owners, as well as their friends and family who may all benefit from using the iPal.

1.1 Requirements List

R-1: Utilize Siri to differentiate and process both information queries and command queries.

R-2: A robust security scheme providing user verification through two-factor authentication.

R-3: The ability to create and modify a user account hierarchy.

R-4: Customizable user access permissions for Siri-enabled applications.

R-5: A permissions configuration web interface (similar to a router's local configuration site).

R-6: A basic framework making use of Siri to interact with Siri-enabled apps from iPal.

R-7: A communication platform to enabling support for Google Home and Amazon web services.

R-8: A machine-learning approach to streamline user interaction with Amazon and Google Home services (similar to the one already built into Siri Web Services).

1.2 Discussion of Requirements

Requirement R-1: For the first requirement, we aim to leverage the existing Siri platform for user command processing. Having this will allow the iPal to take in voice data from a user and directly convert it to text that can be analyzed to determine what the user wants to do. Since this technology is already mature, we feel that it is the right choice to reuse it rather than construct our own speech processor from the ground up. Having this first requirement grants the iPal the ability to determine and then choose how to best service a request from the user.  

Requirement R-2: We propose the following two-factor authentication scheme:

1.     The first factor is the name of the user from the incoming voice data and matching the corresponding username.

2.     We achieve the second factor from identifying the user's unique voice pattern. If a match is made between username and voice pattern, the user is then granted access.

Requirement R-3: The iPal will require a security structure for users consisting of one single administrator and customizable number of guest users. The administrator account will be created during the initial device setup. After this, the administrator can proceed with adding guest user accounts. The idea of this requirement is to allow for the flexibility of more than a single user to use the iPal such as in a family situation, but to maintain the integrity of any sensitive data linked to the administrator's account.

Requirement R-4: Guests need to have customizable (by administrator) access to Siri-enabled apps. Since our handling of Amazon and Google Home web service does not extend support to all of their respective specific apps, we simply leave user access to these two platforms as either all or nothing at the discretion of the administrator. Having this ability provides flexibility in choosing which guests get access to what apps.

Requirement R-5: Setting account permissions one user at a time can be a cumbersome task. Rather than rely on creating and modifying account permissions solely through the use of voice commands, we will provide a web interface to administrators that is accessible from the local network.

Requirement R-6: We already have the framework to communicate with any Siri-enabled app with Siri Web Services. To leverage Siri Web Services we will need to design and develop a communication framework in iPal to work with Siri Web Services. Siri Web Services will identify the parts of the query from the user's spoken words and categorize the words to determine whether what the user desires is an informational query like "What is the temperature outside?" or a command query such as "Turn off my bedroom lights." Siri Web Services already has the keyword recognition to know how to process both types of queries. It also has a machine learning ability to speed up the interactions with Siri enabled apps over time. Again all we will need to design and develop is the communication framework to get the spoken words from the user to Siri Web Services.

Requirement R-7: We will not limit the ability to control apps to just Siri-enabled apps.  We will support ordering-related applications available through Amazon Web Services, as well as Google Home Web Services applications. By offering support for all three major hands-free assistant platforms, the iPal is poised to attract users from these competing platforms. We will need the communication framework in iPal's software system to support both Amazon and Google Home-related functions.

Requirement R-8: Although our processing of Siri-related commands benefits from Siri's robust machine-learning technology, we are not lucky to have such features made available to us in processing Amazon or Google Home commands. We must develop a separate machine-learning framework to simplify interactions with these two platforms.
